What we're actually building

A single AuraVault smart contract per user. It owns one Mezo Trove on the user's behalf and exposes four operations: openVault, addCollateral, repayDebt, closeVault. v2 adds auto-routing of idle MUSD into Mezo's MUSD Savings Vault and a keeper that uses harvested yield to chip away at debt principal — so the BTC collateral "pays for itself."

Cut from the original pitch after the capability audit (docs/research/mezo-validation.md): ERC-4337/passkey UX (no AA on Mezo), Wormhole NTT MUSD bridging (we don't own MUSD), "$50 freelancer rent loan" use case (1,800 MUSD debt floor too high), real-BTC bridge demo on testnet (mainnet only).

Crypto necessity check

What breaks if you remove the blockchain? Everything. Troves, MUSD minting, BTC collateral locks, on-chain liquidation — all require Mezo. There is no off-chain version of this. Necessary, not ornamental.

Integration-first assessment

AuraVault.sol is already a thin wrapper over BorrowerOperations. The v2 yield routing layer integrates Mezo's first-party Savings Vault rather than building custom yield strategy logic. The only custom code is the auto-repay keeper logic — which is itself just timed calls to repayMUSD. Healthily integration-first. Do not pivot toward writing a custom CDP — Mezo already has one and any custom version would lose the audit time savings.

Next steps

Before any more code: spend two days finding 5 Mezo testnet users with active Troves. Ask "how are you managing your position today?" Listen for emotion, specificity, and workarounds. Vague interest is not demand. If you can't find 5 talkable users, that's itself a signal — pivot toward a dev-tool / SDK angle instead of a consumer product.
  1. User research: 5 Mezo Trove holders via @MezoNetwork Twitter replies + Mezo Discord. Two days, no code.
  2. Apply to Mezo Alpha Builder Program and Supernormal Foundation — there is real ecosystem money available.
  3. Position pitch explicitly against TrovePilot: AURA = borrow + yield automation for individual users; TrovePilot = protocol-side liquidation keepers. Don't let judges conflate them.
  4. Integration test on Mezo testnet — open a real Trove via AuraVault. Confirm BorrowerOperations ABI matches our interfaces before week-2 work.
  5. Ship frontend BEFORE the keeper. User research needs a tangible thing to react to. Recommend a Next.js PWA reading from Wagmi over the deployed vault.
  6. Keep cutting: no AA, no Wormhole NTT, no in-network invoices, no cross-chain settlement until v3 or never.

Pivot suggestion (only if user research fails)

If 2 days of outreach yields no usable conversations, the retail thesis is wrong for now. Pivot: package AuraVault + interfaces + a TypeScript SDK as a Mezo developer kit for teams building on top of MUSD. Same code, different ICP. Apply to the Alpha Builder program with that framing.
